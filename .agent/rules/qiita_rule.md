# AIを活用した技術ナレッジプラットフォーム向け最適化記事生成・ドキュメント作成要領

本ドキュメントは、Qiita等の技術共有エコシステムにおいて、AI（大規模言語モデル）を用いてエンゲージメント（インプレッション、いいね、ストック）を最大化しつつ、プラットフォームのガバナンスと規約を遵守した高品質な記事・ドキュメントを生成・作成するための包括的なルールセットである。

## 1. 共通執筆・生成ルール（修辞的・構文的最適化）
記事の透明性を高め、読者の「認知負荷（Cognitive Load）」を最小化するために、プロンプト等で以下の原則を強制する。

- **一文一義と短文の原則**: 1文あたりの文字数を原則60文字〜80文字以内とし、1つの文に複数の意味や接続詞を過剰に詰め込まない。
- **アクティブボイス（能動態）の徹底**: 「〜が設定される」のような曖昧な受動態を避け、「システムが同期を実行する」「ユーザーがパラメーターを設定する」など、動作の主体を明確にする。
- **専門用語の統制とコンテキストの提供**: ターゲット読者のレベルを意識し、初出の難解な用語や略語には必ず短い定義や言い換えを同封する。
- **例や比喩（メタファー）の先行提示**: 抽象的で複雑な概念・アーキテクチャを説明する際は、技術仕様を語る前に現実の物理法則や業務プロセスに例えた直感的な比喩を提示し、理解の「足場（Scaffolding）」を作る。
- **簡潔な表現と常体**: 回答および主張は簡潔に表現し、常体（だ・である調）で事実を記載すること。箇条書きを使用する場合、文末の句点（。）は一切使用しないこと。
- **視覚的整理とMarkdownの活用**: 複雑な条件や3つ以上の要素が並ぶ場合は、Markdownの箇条書きや表を活用し、スキャニング（拾い読み）しやすい構造を維持する。成果物のコードは必ずコードブロックで出力する。

## 2. 記事自動生成パイプライン（The Generation Pipeline）
AIによる記事生成は、プロンプトによる一括生成（ゼロショット）ではなく、「企画・構成」「執筆・推論」「検証・校正」の独立した3フェーズ設計（マルチステップ・アーキテクチャ）とする。

### フェーズ1：コンテキスト定義とメタデータ設計（企画・構成）
- **結論先行型のアウトライン生成（トップダウン構造）**: まず記事の構造（目次）のみをAIに生成させる。一般的な「起承転結」を排除し、【結論（システム構成図や完成図等の成果物） → 理由・背景 → 具体的な実装・手順 → 落とし穴と対策 → ネクストアクション】の順に配置してROIを事前提示する。
- **5W1Hと前提条件の明示**: 記事の冒頭に、実行環境やバージョン（When/Where）、対象読者（Who）、最終的な解決状態（What）、アプローチの選定理由（Why）、ステップの概要（How）を網羅したサマリーを強制的に生成させる。
- **読者の関心を惹くタイトル**: 権威性や実績、あるいは意外な問いを含める。「スクラッチ構築」等の専門用語を適切に用い、「絶対に壊れない」等の誇張表現は控える。

### フェーズ2：認知的最適化とテキスト生成（執筆・推論）
- **心理的インタラクションとペルソナの共感**: 読者が直面する問題（ペインポイント）を想定し、「なぜこのエラーに遭遇したか」「ここでつまずきやすい」という失敗のナラティブや、読者への擬似的な対話を各主要セクションに組み込む。
- **読者層に合わせたトーンの動的調整**:
  - 初心者（Beginner）向け：メタファーを多用し、心理的ハードルを下げる励ましのトーンを打ち出す。
  - エンジニア（Engineer）向け：過度な情緒的表現を避け、限界事例（エッジケース）の考察、パフォーマンスの定量的データ、トレードオフの冷静な分析を重視する。
- **論理的背景の解説**: 構成の中盤では、単なる手順の羅列にとどまらず、技術的・論理的な背景（例：なぜその設定が必要か）を解説し、可能なら他環境との比較表を挿入する。

### フェーズ3：事実検証とプラットフォーム適合性チェック（検証・校正）
必ず人間の介入（Human-in-the-loop）または独立した検証用AI（Critic要素）を通し、コミュニティ破壊や規約違反のリスクをブロックする。

- **ファクトチェックとハルシネーションの排除**: 記事内で紹介したライブラリ、関数コマンド、公式ドキュメントURLの実在と正確性を検証する。非推奨（Deprecated）メソッドが混入していないか確認する。
- **詰まりポイントとネクストアクションの明記（必須）**: 作業中の「詰まりポイント（落とし穴、エラー、トラブルシューティング）」とその具体的な対策を必ずセットで記録し、最後に読者がすぐ試せる次のアクションを提示する。
- **セキュリティ・情報漏洩の多重フィルタリング**: プロンプトに混入した社内コード、プライベートリポジトリURL、認証トークン、AWSアクセスキー、個人情報などが生成結果に含まれていないか、正規表現とAIの文脈理解で厳格にマスキングする。

## 3. リスクドメインと防御策（プラットフォームガバナンス要件）
Qiitaガイドライン改定（「AIが生成した内容は正確性を確かめよう」等）および利用規約に基づき、以下の防御システムを実践・実装する。

| リスクドメイン | AI自動生成特有の脆弱性および課題 | プラットフォーム規定とのコンテキスト | 実装すべき技術的・プロセス的防御策 |
| :--- | :--- | :--- | :--- |
| **情報の正確性と信頼性** | ハルシネーション（幻覚）による架空の仕様やコードの捏造 | ガイドラインにおける「正確性の確認」の義務化、結果責任の投稿者への帰属 | 生成されたコード片のサンドボックス自動実行テスト、公式ドキュメントURLの自動クロスリファレンス検証 |
| **アカウント保全とスパム判定** | 高頻度かつ低品質なテキストの自動連続投稿によるエコシステムの汚染 | 利用規約第1条などに基づく、違反アカウントの即時不可逆的な凍結措置 | 投稿間隔のレートリミット設定、生成テキストの独自性（Perplexity）や内容重複のスコアリング・フィルタリング |
| **セキュリティと情報漏洩** | プロンプト入力時の不注意による機密な社内構成やAPIキーの露呈 | プライバシーポリシー第17条の遵守および一般的な情報セキュリティ要件 | 入出力データに対する正規表現を用いたPIIマスキング、シークレットスキャンツールとの統合 |

## 4. 技術メモ作成要領（開発フェーズ引継ぎ・備忘録用）
記事化する前の段階や、特定のフェーズにおける基本動作検証を「技術メモ」として社内・個人向けに蓄積する場合の要件である。

- **結論とDoD（完了定義）**: 冒頭でメモの目的と、達成すべき具体的な完了状態を定義する。
- **比較検証の可視化**: 複数の手法やツールを検討する際は、表を用いて横並びで比較する。
- **UIと基本操作の解説**: 対象ツール（ComfyUIなど）の画面操作方法を直感的に理解できるよう箇条書きでまとめる。
- **コンポーネントの役割**: ワークフローを構成する各要素の役割を平易な言葉で一つずつ分解して解説する。
- **ステップバイステップの検証手順**: 読者（または自分）が迷わず実行できる具体的な手順を番号付きリストで記載する。
- **次のアクション（検証後）**: 検証完了後に実施すべき内容を明確に提示する。
